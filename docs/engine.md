# Lucid — Behavioral Engine Specification

**Date:** 2026-07-03 · **Status:** Canonical — a living concept, evolving with the project
**Scope:** The Engine subsystem defined in [`docs/architecture.md`](architecture.md) §3. This document specifies runtime mechanics, telemetry, accountability, and governance. It contains no instance data; user calibration lives in `personal/calibration.md` (guide: [`docs/calibration.md`](calibration.md)).

## 1. Definitions

**Chain** — an ordered sequence of Links anchored to a Bell. **Link** — one behavior in a Chain. **Floor** — the minimum version of a Link or Chain that still counts as completion; floor completion equals completion for all adherence math. **Bell** — the external trigger that initiates a Chain; the user never initiates from memory or motivation. **Dispatch** — the routine (initially just the Bell plus a fixed template; eventually an agent) that reads the Charter, the calendar, the declared mode, and the avoided-task queue, and names the day's allocation: today's Crux and the Chain at its declared level. The Dispatch never invents new obligations. **Config** — a one-time or ambient change requiring no ongoing willpower (a staged object, a removed app, a fixed supplement anchor); Config items consume no runtime slot. **Opportunity habit** — a trigger-based, slotless practice executed when its cue naturally occurs; logged as a tally, never quota'd. **Crux** — the day's single most-avoided live task. **Gate** — a scheduled checkpoint (default: days 30/60/90) where Chain additions are earned. **Retro** — the weekly review; the only place runtime parameters change. **Hearing** — a Parked item's scheduled quarterly trial. **Away Mode** — a named, pre-specified travel configuration of a Chain. **Storm** — a bounded, witness-confirmed incapacity state during which contact continues and the stake is stayed (§4). **Ambush window** — a calendar-known hard period named in the Charter in advance (an anniversary, a deadline season); enters the storm state automatically. **Profile** — a named clock configuration (Bell time, tripwire hour, rollover boundary) for a recurring schedule; a profile switch moves all three together (§2). **Logical day** — calendar day after applying the active profile's rollover boundary (default 04:00): events before the boundary attribute to the previous day. A close-out belongs to the Bell that initiated it: after the boundary, if the previous logical day has no day record and the current day's Bell has not yet rung, the close-out attributes to the previous logical day; `/closeout today` forces current-day attribution.

## 2. Runtime

**Priority order (fixed, non-negotiable): chain > record > analysis.** If tooling breaks, a text message to oneself counts as the record and gets backfilled later. Losing a night of data is recoverable; losing the chain is the actual incident (architecture P10). Backfill is a first-class path: `/closeout backfill` creates or corrects a recent day's record (window: 7 days, §8) as an appended correction folded at read time — derived state recomputes, sent messages are never unsent ([`docs/mvp/engine-module.md`](mvp/engine-module.md)).

A Chain executes at its Bell. Each Link's completion cues the next; the Chain compresses to its Floor under load but never skips — floors exist precisely for the worst days. Chains are deliberately scarce: the runtime holds one Chain until the first Gate, and additions are earned, never enthusiasm-driven — earned in ordinary weather, not in survival (§5). **The total daily runtime footprint stays under a fixed cap (default ~30 minutes) plus whatever rides existing Anchor slots.** Gates may add Links only within the cap; a Link that would exceed it must displace something. This cap is what structurally keeps the runtime tiny — structure at the anchor points, freedom everywhere else.

New Links are wrapped in an existing Anchor identity wherever possible (a mobility practice belongs to the sport it serves), because identity-wrapped practices bypass the direct-pursuit failure mode.

**Baseline week.** Week 1 of any new Chain is calibration: run the chain, capture everything, change nothing. No parameter changes at the first Retro except lowering a floor. You cannot measure improvement without a baseline. One rehearsal is part of it: **declare Yellow once, deliberately, on a day of your choosing** — a drill, not a response to load. A Yellow floor scores 1.0, so the drill costs nothing, and the first real hard day finds the lever already warm. Relief valves that are only read about get discovered mid-crisis or never; the ones that get rehearsed get used.

**Operating modes.** Each day carries a declared mode: **Green** (full Chain plus Crux), **Yellow** (floors and Config only), **Red** (survival floor: the single smallest countable act). Undeclared days default to Green. Adherence is always measured against the declared mode — a Yellow day executed at Yellow scores 100%. Modes make enforcement fair enough to be legitimate: a push that ignores load gets ignored; a push calibrated to declared capacity has standing.

**Floor semantics, reconciled.** Floor completion is completion for all teeth math — streaks, misses, budget, escalation — and mode-relative adherence also scores a floor at 1.0: a Yellow floor was the declared plan, and a Green floor still kept the chain alive. Floors must never become shame math; that is the design's soul. The honest counterweight is structural, not punitive: every surface that shows adherence co-presents the floor-day ratio and raw days-accounted — an honest pair of co-numbers, never a score, never witness-visible.

**Mode declaration timing.** A mode may be declared or downgraded at any point up to the day's Bell; after the Bell it is fixed for that logical day. Retroactive declaration or amendment is prohibited — the mode ledger is append-only like everything else. A hard day that went undeclared is exactly what the isolated-miss budget exists for; the Retro may annotate context on a miss but never rescores it. This asymmetry (easy to declare honestly beforehand, impossible to renegotiate afterward) is what keeps modes from becoming the escape hatch.

**Schedule profiles.** A rotating roster or a split life gets profiles, not exceptions: named clock configurations — Bell time, tripwire hour, rollover boundary — defined in the chain configuration and amended only at a Retro (P8). The mode-declaration deadline follows the active profile's Bell automatically. *Switching* between defined profiles is not a parameter change; it is declaring reality, like declaring a mode: sticky until the next switch, recorded append-only, and **effective from the next logical day, never the current one** — a switch can never dodge a bell that has already rung, and the outgoing day completes under the clocks that started it. Switch frequency is Tier-3 telemetry reviewed at Retro: churn is a canary, not a sin. Profiles move clocks; Away Mode (§5) compresses Links — orthogonal, and they compose. A fixed clock was never a design commitment; it was an assumption the night shift disproved.

**Capacity signal.** One digit (1–5) captured nightly — what did today's body and mind allow — with an optional one-word limiter tag. **Polarity: 1 = nothing available, 5 = fully resourced.** Scale meaning: 1 — survival only; 2 — floors were a reach; 3 — ordinary; 4 — resourced; 5 — surplus. Tags may be opaque labels; the analyst needs correlations, not diagnoses. The capacity series with mode declarations is exportable as longitudinal context for professional care. The capacity digit is the Engine's *only* body signal — it exists for mode fairness. Richer health observation (pain series, symptoms, intake) lives Mirror-side in [`docs/observations.md`](observations.md), where it is inventory, never obligation; the two series join only in user-invoked projections like the clinician packet.

**The Crux protocol.** Each day, the Dispatch names exactly one Crux and it receives ten minutes of mandatory contact in a fixed slot placed inside the user's peak-energy window — contact, not completion; the obligation is to touch the avoided thing, and finisher dynamics handle the rest more often than predicted. The avoided-task queue carries a WIP limit (default 3). **Queue ingestion:** items enter the queue only by explicit user nomination, which is an Engine event. The Mirror may notice avoidance-shaped material in reflections and offer it as a candidate through the resonance gate — but on acceptance it is the user who places the item in the queue; the Engine reads only the queue and never Mirror content (architecture P3). The weekly Retro runs the avoidance sort on every queue item: **Crux or retire** — some avoided items are growth edges deserving contact; others are misaligned obligations to be formally retired with a written reason. Avoidance is data; the sort is the skill.

## 3. Telemetry

**Tier 1 — Active capture.** The nightly close-out: link completions, mode, capacity digit and tag, and a journal body with a floor of one line (spoken or typed). Hard budget: two minutes. The Tier 1 schema is frozen between Retros; any field addition must displace something. The close-out is self-serve — completable with no AI in the loop (architecture P9); its journal body flows to the Mirror as an ordinary capture, where the sanctuary rule applies to every word of it.

**Tier 2 — Passive capture.** Unbounded and free: timestamps, entry-time series, screen and pickup metrics where the platform provides them (the objective ground truth for the phone habit, not memory of it), device charge events, wearable data if a device is already owned (never buy gear for telemetry), calendar covariates, journal length. No passive source may ever require nightly user effort. The Mirror-side generalization of this tier — enrichers appending weather, daylight, and calendar frame to every logical day — is specified in [`docs/observations.md`](observations.md) §5 and shares the same rule.

**Tier 3 — Derived.** Streak mathematics, rolling adherence per Link, completion-time drift, floor-day ratio, miss clustering by weekday and covariate, error-budget burn. Drift, rising floor ratio, and shrinking entries are leading indicators — canaries reviewed at Retro before they become misses. The floor-day canary carries a number: a floor-day ratio above 40% over the trailing 14 days (§8) trips it and makes a template-framed environment-or-simplification question a mandatory agenda item at the next Retro — a question, never a score, never witness-visible.

## 4. Accountability

**SLO.** Zero consecutive misses (hard), and an isolated-miss budget per 30 days (default 4). A miss followed by a completed night is budget spend, not failure. Two consecutive misses is a breach.

**Escalation ladder.** **L0** — ambient state on a surface the user cannot avoid (prompt, status line, lock screen, physical marker at the point of performance). **L1** — automated private nudge the morning after a single miss. Fixed content: last night was a miss; never miss twice; tonight is a must; the floor version, named explicitly. **L2** — automatic escalation to the Witness after two consecutive misses, via dead-man semantics: the absence of expected completion events fires it; self-report is never the trigger. **L3** — the pre-committed stake, defined in the Charter while strong, executed mechanically on breach or a blown monthly budget, without renegotiation. Each level exists so the level above rarely fires. L1, L2, and their storm variants each end with the same fixed line — *"— the form letter, pre-committed at Day 0."* — because the escalation that lands as a form letter gets a floor run tonight, and the one that lands as a judgment gets an uninstall; the template now says which it is.

**Restart ritual (after any miss or breach).** The floor chain, that night, no makeup work. Nothing is added, nothing is owed, nothing is compensated — the what-the-hell effect is absorbed by making re-entry the smallest possible act. After an L3 breach the stake executes, the restart ritual still applies the same night, gate clocks continue from the original chain start, and the error budget refills on its normal 30-day window. A breach costs the stake; it does not reset the system.

**Tripwire automation.** L1 and L2 must be wired, not intended: escalation runs on schedule without the user's mood participating. Tooling for the tripwire (and everything else in Phase 0) carries a hard **one-day build timebox** — the tool becoming the project is a named failure mode (§6). A system with an unwired escalation path is a suggestion.

**Witness contract.** Witness-facing views expose streak, declared mode, escalation state, and storm state only — computed exclusively from Engine events. Journal content, capacity payloads, and Profile data are never visible at this layer (architecture P3). The witness is briefed once, at Day 0, with a job description of four lines: when L2 fires, ask about it, once, in person or by message; when a storm is declared, confirm or decline it with one line, against the Charter clauses they have seen (see The storm protocol below); when a stake executes, verify with one line that execution occurred (see Stake execution below); and hold the written stake — they are shown it at the briefing and confirm it, because a stake the witness has never seen cannot be executed without renegotiation. The witness channel also receives a **monthly heartbeat** — a fixed present-state snapshot (streak state and whether an escalation is open — never a month summary), same template discipline and consent record as L2; in Phase 0 the user sends it themselves, template verbatim (templates: [`docs/mvp/engine-module.md`](mvp/engine-module.md)). It exists so the briefing's backstop works: *total silence — a full calendar month with neither an L2 nor a heartbeat — is the witness's cue that the tooling died: ask once.* This is the dead-man switch for the dead-man switch: if the tripwire tooling silently dies, the system still has exactly one human check it cannot lose. The brief carries one further line, licensed at Day 0: if an L2 goes unanswered and total silence persists for 7 days (§8), the witness may — as a human, outside the system, in their own words, without Lucid framing — reach out once more. Granted at Day 0, the message arrives expected, not surveillant; this is briefing text, not a send path, and the message ceiling is untouched.

**Witness lifecycle.** The brief is re-confirmed quarterly, folded into the stake review — one message reconfirming the contract, recorded. A witness may resign at any time. On resignation or sustained unreachability the system enters **witness-lapsed** state: `l2_enabled` goes false, the ladder degrades to L1-only, and `/status` surfaces "witness lapsed — L2 disarmed" until a replacement completes the full Day-0 witness flow (briefing, stake shown, confirmation, channel scoping). Every lifecycle transition appends to a `status_history[]` in the witness contract record ([`docs/mvp/engine-module.md`](mvp/engine-module.md) `witness.json`).

**Stake criteria.** Concrete, mildly painful, and mechanically executable: a payment to a cause the user dislikes, a specific owed obligation — decided while motivated, written into the Charter, triggered without discussion. A stake that requires deliberation at execution time is not a stake.

**Stake execution.** The executor is the user; the verifier is the witness. On a breach or a blown monthly budget, the user executes the stake within the stake execution window (default 72 hours, §8) and confirms to the witness, who verifies with one line that execution occurred — part of the Day-0 brief, not a job invented at breach time. Past the window, the system enters `stake_owed` state: surfaced on `/status`, and Gates cannot ratchet (simplify or hold only) while a stake is owed. No new sends exist anywhere in this path. **Waiver** is never unilateral and never silent: the user proposes, the witness confirms, a written reason is recorded, and all waivers are reviewed at the quarterly stake review — a stake that keeps getting waived is a miscalibrated stake, and the review's job is to replace it, not forgive it. **Re-arm** is immediate after execution or a recorded waiver; stake definitions themselves change only at the quarterly review.

**The storm protocol.** Modes absorb ordinary variance, declared day by day before the Bell; the storm protocol is the extraordinary instrument — for the hospitalization, the bereavement, the collapse that arrives without warning and stays longer than a day. It is the valve that keeps the teeth pointed at avoidance, the only thing they were ever for: the machine must never again fine a person for grieving, getting hurt, or working the night shift. Every element is pre-committed; nothing about it is negotiated in the moment.

*Entry* — three paths, all recorded append-only, all applying from the declaration **forward, never backward** (yesterday's miss cannot be re-filed as weather):

1. A **storm clause** — conditions written into the Charter at Day 0, while strong, in the user's own words — is invoked by declaration and stands only when the witness confirms it with one line, against the clauses they have seen. An unconfirmed declaration lapses after the confirmation window (§8).
2. An **ambush window** — a calendar-known hard period named in the Charter with dates — enters automatically; it was confirmed when the witness saw the Charter.
3. An **unwritten storm** — life exceeding the enumeration — may be declared and witness-confirmed exactly like a clause; writing the missing clause becomes a mandatory agenda item at the next quarterly review. What is pre-committed is the protocol and the confirmer, not an exhaustive catalogue of catastrophe.

*While standing.* Undeclared days default to **Red** instead of Green — the floor is the ask, without nightly ceremony; a declared mode still wins and still fixes at the Bell. Day records are annotated, never rescored: a missed night records as a miss, the streak breaks honestly, and the annotation says why. The Bell keeps ringing (P7), and the tripwire keeps contact: L1 and L2 fire on their ordinary schedule using fixed **storm variants** of the same templates ([`docs/mvp/engine-module.md`](mvp/engine-module.md)) — same send classes, no new consent; contact is a feature, because what a person in collapse needs is the knock, not the fine. What changes is the consequence arithmetic: storm misses spend no error budget and accrue no breach — **a storm miss is never a stake event**, not deferred, never — and Gates hold (no ratchet) while a storm stands. The witness's topline view carries the storm state; they co-signed it. The Mirror is untouched: nothing here reads, scores, or exposes a word of content (P3).

*Bounds and review.* A storm stands for a fixed duration (default 14 days, §8), renewable once by the same declare-and-confirm act; expiry is automatic, and the user may end it early by declaration. Breach math resets at exit — the first post-storm miss is an isolated miss, not the third act of the storm. Every entry and exit is an input to the next Retro, and storms are reviewed at the quarterly: more than the threshold per quarter (§8) makes recalibration a mandatory agenda item, because a storm clause doing the chain's job means the chain is mis-sized or the season needs the Parked treatment. A review question, never a penalty.

## 5. Governance

**Weekly Retro.** Twenty minutes, fixed slot. Inputs: the week's three telemetry tiers, any storm entries or exits (§4), plus one optional **excavation block** — a single historical entry (a formative moment, an old pattern, a piece of relationship history) captured to the Ledger with its correct `occurred_at`, feeding the Mirror's recontextualization. Output: at most one change, selected by pre-committed rules — adherence under 5/7 → simplify (lower a floor, move the Bell earlier, remove friction), never intensify; fourteen consecutive clean days with stable timing → eligible to ratchet one parameter or hold; entry-time drift beyond 45 minutes across the week → move the Bell to match reality; a canary trip without a miss → environmental fix only. One change per iteration keeps week-over-week comparison causally readable. When an experiment is open ([`docs/scientist.md`](scientist.md)), the Retro is also its block boundary: arm switches happen at a Retro and only there — the one-change rule gives block designs for free. Baseline week (§2) permits simplification only.

**Missed Retro.** A week with no Retro freezes parameters — no changes of any kind — while telemetry accrues; the next Retro covers the gap. Two consecutive missed Retros trip a governance canary surfaced on `/status` — L0 ambient only: governance is not a Link; it never escalates and never reaches the witness. And because a spec full of floors owes its own governance one, a **floored Retro** exists: five minutes — read the canaries, choose exactly one simplification or hold. It counts as a Retro.

**Gates.** At days 30/60/90 from Chain start: ratcheting — attaching one new Link or activating one Parked item — requires rolling adherence at or above the threshold (default 85%) with stable timing, **and** a trailing-30-day floor-day ratio at or below the canary threshold (§3), **and** Yellow-declared days at or below 50% of the gate's trailing 30 days (§8). When any condition fails, the gate's only outputs are hold or simplification — never penalty. Chronic Yellow is explicitly legitimate: it survives, it keeps streaks alive, it simply does not earn additions — additions are earned in ordinary weather, not in survival (§2). A gate cannot ratchet while a stake is owed (§4). Additions must fit inside the runtime footprint cap (§2). Physical-loading Links additionally require any registered clinical gates in the instance configuration to be satisfied. Experiment verdicts are delivered at Gates and enter the resonance gate as proposals, never as changes ([`docs/scientist.md`](scientist.md)). Day 90 is a full postmortem: what graduates to permanent, what gets dropped, what the next 90 looks like.

**Quarterly.** Charter review with reasoned amendments; Hearings for every Parked item; framework-stack changes; stake review; storm review (§4) — including writing any clause an unwritten storm exposed.

**Away Mode.** Defined before travel, as configuration: which Links compress, what the Floor becomes, which capture surface substitutes. Floor nights in Away Mode count as completions; the error budget remains active. Travel is an operating mode, not an exception negotiated in the moment.

**Day 0 — definition of done.** The program starts when, and only when, all of the following exist: the Bell is set with the Chain named in its label; the environment is staged (each Link's point of performance prepared); the witness is briefed and has confirmed; the stake is written and shared with the witness; storm clauses and ambush windows are drafted in the Charter and shown to the witness (they may be empty; they must be chosen — an unchosen valve is how teeth end up biting circumstance); the Ledger location is chosen; the off-limits registry is drafted (it may be empty; it must be chosen); Away Mode for any known travel is acknowledged; a schedule profile exists for each recurring roster, if the schedule rotates; **the comprehension pass is done** — the user has read the actual L1, L2, storm-variant, and heartbeat template texts verbatim (consent granted in advance, P5, means knowing the exact words that will arrive) and walked the five relief valves once (floors, modes, budget, backfill, storm — [`docs/calibration.md`](calibration.md)), with the baseline week's Yellow rehearsal (§2) on the calendar; and the first weekly Retro and first quarterly Hearing are on the calendar. Total setup is timeboxed to one sitting for ignition (the Bell and the environment — five minutes, tonight) and one weekend for the rest. None of it is optional: an unwired escalation path is a suggestion, and suggestions have a documented track record.

## 6. Failure modes and mitigations

| Failure mode | Mitigation |
|---|---|
| Never actually starts (ignition failure) | Bells, staged environment, floors — starting is the designed-for case, not the assumed one. |
| The novelty cliff (~day 12) | Floors plus the L1 tripwire; the chain compresses, it never skips. |
| The what-the-hell effect after a miss | Never-miss-twice plus the restart ritual: floor chain, that night, no makeup work. |
| Configuration without comprehension (relief valves discovered mid-crisis, or never) | The Day-0 comprehension pass (§5); the baseline week's Yellow rehearsal (§2); templates that self-identify as pre-committed form letters (§4). |
| Protocol thrash | Frozen Tier 1 schema; one change per Retro; baseline week. |
| Measurement fatigue | Passive-heavy telemetry; two-minute active budget; fields must displace to enter. |
| The tool becoming the project | One-day tooling timebox; chain > record > analysis; the cathedral clause (architecture §8, Phase 2). |
| Modes becoming the escape hatch | Declaration fixed at the Bell; no retroactive amendment; misses absorbed by budget, not renegotiation. |
| Escalation firing on circumstance, not avoidance | The storm protocol (§4): witness-confirmed, bounded, stake-staying, never retroactive; ambush windows pre-named at Day 0; contact preserved throughout. |
| The storm becoming the escape hatch | Witness confirmation against pre-written clauses; fixed duration, one renewal; quarterly review threshold; entry never retroactive; the honest record still shows every miss. |
| Travel and schedule chaos | Away Mode as pre-specced configuration; schedule profiles moving Bell, tripwire, and rollover together (§2); the rollover boundary and close-out attribution for late nights (§1). |

## 7. The AI layer

AI is the analyst, never the database (architecture P6, P9). The Ledger is a durable artifact the user owns; models read excerpts, never solely hold the corpus. Sanctioned uses: the weekly Retro copilot (miss-clustering, drift detection, covariate analysis, and a pressure-test of the one proposed change), deep analysis at gates, automation of the L1/L2 tripwire, and — on the Mirror side, at the user's invitation — reflection over the journal stream under the sanctuary rule. What AI is not for: being a dependency of the daily chain, motivation, or being the place the data lives.
